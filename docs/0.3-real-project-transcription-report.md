# MimiSpec 0.3 Real-Project Transcription Trial

> Date: 2026-07-20
>
> Status: technical usability evidence supporting a `0.3.0-dev` development
> snapshot. This is not an independent-author trial, does not satisfy the
> 5-author/25-document release gate, and cannot justify `0.3.0-rc.1`.

## Conclusion

The current 0.3 working tree is usable for extracting, preserving, reviewing,
and structurally organizing intent from four non-trivial MIMI projects. It is not
yet a complete production bridge from existing MIMI code to confirmed MMS,
target materialization, or Evidence.

| Area | Trial result |
|---|---|
| Plain-language and structured transcription | Usable for all four projects |
| Canonical parse | Complete, zero errors after syntax adaptation |
| Lossless source preservation | Byte-exact |
| Normalized render and reparse | Equivalent semantic AST |
| Rule attachment | No pending/dropped attachment diagnostics |
| Commitment workflow | Correctly produces review queues and zero automatic locks |
| Large-document review ergonomics | QueueTree grouping, editor navigation, and atomic Human batching implemented |
| Mimi Profile/materialization | Prototype only; no confirmed slots were eligible |
| `0.3.0-dev` readiness | Ready as a labelled technical snapshot candidate |
| `0.3.0-rc.1` readiness | Blocked; external trial remains incomplete |

No silent parser data loss was found in these four documents. The second round
also showed that implementation bugs can be recorded as reviewable residuals
without accidentally promoting them to confirmed product intent.

## Source projects

The trial now covers four projects from the sibling MIMI checkout:

1. `projects/mimi-kv/src/main.mimi` and `client.mimi` — a TCP key-value server
   and REPL client with a line protocol, mutable map state, JSON persistence,
   file/socket failure paths, and 265 total source lines.
2. `projects/mimichat/src/main.mimi` — a 2,014-line multi-user chat system with
   per-connection tasks, `UserManager` and `RoomManager` actors, room/private
   messaging, nickname changes, history persistence, CLI client, and an
   embedded acceptance suite.
3. `projects/mimi-markdown/main.mimi` — a 1,009-line Markdown parser and HTML
   renderer with cursor utilities, inline formatting, lists, fenced code,
   blockquotes, tables, path helpers, and output-oriented tests.
4. `projects/mimi-log/src/main.mimi` — a 755-line log-analysis pipeline for
   Apache/Nginx/JSON Lines parsing, composable filters, latency percentiles,
   grouping, CSV/JSON output, CLI options, and file-reading demonstrations.

The resulting review documents are:

- [`corpora/mimi-kv-real-project.mms`](corpora/mimi-kv-real-project.mms)
- [`corpora/mimichat-real-project.mms`](corpora/mimichat-real-project.mms)
- [`corpora/mimi-markdown-real-project.mms`](corpora/mimi-markdown-real-project.mms)
- [`corpora/mimi-log-real-project.mms`](corpora/mimi-log-real-project.mms)

They contain 131, 221, 217, and 257 lines respectively. MIMI implementation
details are not copied into the Core AST. Runtime behavior, security boundaries,
parsing policy, persistence, concurrency, serialization, and observed defects
are represented through natural-language rules, Flows, functions, and residual
questions.

The implementation is treated as observed input rather than human authority.
Consequently the transcription uses `?` for inferred intent and creates no
`$`/`$$` commitments.

## Automated contract

All four files are embedded into `corpus_acceptance_tests`, which asserts:

1. complete semantic parsing with no errors;
2. byte-exact lossless rendering;
3. normalized render/reparse AST equivalence;
4. `mimispec.ast/0.3` JSON serialization;
5. preservation of exact type, Flow, function, module, semantic-slot,
   decision, and delegation counts;
6. zero automatic commit-ready slots;
7. no diagnostics outside the expected decision/delegation queues.

Observed structure and collaboration results:

| Document | Types | Flows | Funcs | Modules | Slots | Decisions | Commit-ready |
|---|---:|---:|---:|---:|---:|---:|---:|
| mimi-kv | 4 | 2 | 7 | 2 | 401 | 48 | 0 |
| mimichat | 4 | 2 | 13 | 6 | 680 | 73 | 0 |
| mimi-markdown | 4 | 2 | 13 | 5 | 649 | 70 | 0 |
| mimi-log | 5 | 2 | 14 | 5 | 752 | 79 | 0 |

The Mimi Profile prototype reports no supported/partial confirmed slots because
every candidate remains under human review. Materialization keeps every
semantic suffix independent and reports open residual slots explicitly rather
than folding a node to its strongest state. This is the correct commitment
result: transcription from code must not auto-confirm human intent.

## Problems exposed

### P1 — Natural action-line syntax is easy to cross accidentally — resolved

The first drafts produced three parser errors in mimi-kv and five in mimichat.
Common prose such as `error.visible`, `non-empty`, `newline-delimited`,
`bind and listen`, and `unregister ... then ...` entered reserved expression or
control syntax rather than behaving as free-form action text. Diagnostics such
as “expected atom” or “expected `:`” did not explain that a `desc` step or a
simpler action phrase was needed.

The documents were made valid by using non-reserved identifiers and splitting
compound actions. This is learnable, but it weakens the five-minute promise for
authors who reasonably treat step lines as natural language.

`docs/specification.md` already documents quoted Atom escaping and standalone
`desc` steps. The trial problem is discoverability and recovery guidance, not
the total absence of an escape mechanism.

The grammar remains unchanged. E0010 now explains reserved-word, embedded
`on`/`desc`, and hyphen collisions. Tier-3 `SyntaxQuickFix` offers either a
quoted action label or `desc "..."`; standard LSP code actions return a
`WorkspaceEdit` after revision-checked `codeAction/resolve` registers the
pending Human transaction. Lines containing transitions, assignments, or real
blocks are guidance-only. Monaco snippets teach the same escape boundary.

### P1 — Review queues need hierarchy and batching — resolved

Faithfully marking AI-inferred structures with `?` generates 48 and 73 decision
items. Slot precision is correct, but the CLI produces a long flat list with no
module grouping, inherited review view, collapse, filtering, or bulk
human-review operation.

The compatible flat arrays remain, while QueueTree groups exact items under
source-order scopes and the VS Code view navigates to them. Human-only
`mimispec/prepareQueueBatch` validates exact current slot IDs and atomically
returns suffix-only edits. Stale, duplicate, unknown, strong-lock, AI, or mixed
illegal batches produce no partial transaction.

### Deferred production gap — Mimi integration is still a prototype

Core could express the product intent without copying MIMI syntax, which is a
positive result. It could not represent or validate Actor mailbox policy,
Capability ownership, runtime task cancellation, atomic persistence, or target
verification as Core semantics—and should not attempt to do so.

A future Mimi Profile/adapter needs source provenance, residual-intent output,
capability probing, generated-target traceability, and independent Evidence.
The current experimental profile only classifies commit-ready top-level slots;
it does not provide this production bridge.

### P2 — Nested intent-gap analysis is incomplete — resolved conservatively

`detect_intent_gaps` currently iterates only top-level semantic fragments.
Functions inside modules—the normal organization used by both transcriptions—
do not receive the function error-path heuristic. This avoids noise today but
means a clean diagnostic report is not proof that nested functions were
analyzed.

The diagnostic now uses parser source-order/kind/scope correspondence instead
of name substrings, recurses nested Contexts, skips Partial documents, and only
warns functions whose steps contain conservative file/network/persistence
boundary terms without an explicit error/on/failure clause.

### P2 — Cross-revision/source provenance is insufficient — experimental sidecar added

All four corpora now have `mimispec.provenance/0.1` fixtures. Their locator combines
scope path, node/anchor/footprint kinds, owner ordinal, and protected SHA-256;
ambiguity never falls back to a revision-local ID. `mimispec provenance check`
restricts all referenced paths to explicit `--source-root` and validates MMS/
source hashes without invoking MIMI or changing commitment. This is still an
experimental sidecar, not a production Evidence ledger or adapter.

### Fixed during this trial — forced-removal Flow path

`RoomMembership` includes `on Kicked >>> Outside`, but diagnostics originally
reported that the Flow had no failure/cancel path. Failure-label recognition
now includes forced-removal terms such as `Kicked`/`踢出`, with a minimized
regression test.

### Second round — defect versus intent separation works

The mimi-markdown and mimi-log transcriptions deliberately distinguish three
classes of information:

1. observed implemented behavior;
2. product rules inferred from names, comments, and CLI descriptions;
3. implementation defects or missing behavior that must not become locked
   requirements merely because they exist in source.

This separation was practical with ordinary `desc?`, `rule?`, Flow, and
function clauses. No target syntax or implementation-specific AST was needed.
In particular, the corpus can say that status filtering currently duplicates
records while separately stating the reviewable invariant that a matching
record must appear at most once.

The new drafts parsed cleanly on the first canonical-parser pass. The main
authoring cost was not grammar recovery but deciding the correct review
granularity: literal helper-by-helper transcription produced too many low-value
slots, while only writing a product summary hid concrete safety and correctness
questions. Grouping by parser, renderer, filter, statistics, serialization,
and CLI responsibilities gave navigable QueueTree scopes with 70 and 79
decisions.

The first diagnostic pass found two genuine omissions: block recognition had
no malformed-input transition, and the boundary-facing log runner did not
state a visible failure outcome. Adding those as still-unconfirmed `?` intent
removed both `H-INTENT-GAP` findings. The final documents have no non-queue
diagnostics.

### Second round — security and malformed-input intent remain natural language

The Markdown source exposes HTML injection boundaries and the log source uses
unchecked split indexing plus hand-built CSV/JSON. MimiSpec can make the desired
safety outcomes and unresolved policies reviewable, but Core does not prove
escaping completeness, parser totality, or output conformance. Those checks
remain tests, target Evidence, or adapter responsibilities after humans confirm
the intent.

### Second round — global CLI flags disabled subcommands — fixed

Running the documented JSON workflow as `mimispec --json diagnose file.mms`
treated `diagnose` as a legacy positional filename, while placing `--json`
after the subcommand worked. The root variadic file argument combined with
Clap's `args_conflicts_with_subcommands` setting made a declared global flag
disable subcommand recognition. Removing that conflict setting restores both
flag orders while preserving `mimispec file.mms --json`; parser-level tests now
cover all three forms.

## Product questions surfaced by transcription

The exercise also exposed unresolved intent in the source projects, rather
than parser failures:

- mimi-kv has no explicit TCP framing behavior for split/coalesced packets,
  multi-command connections, or oversized values;
- mimi-kv does not state atomic-save, crash-consistency, concurrent-client, or
  multi-process persistence policy;
- mimichat configuration contains `max_clients`, but enforcement intent is not
  evident;
- mimichat KICK behavior lacks an explicit administrator/authorization model;
- mimichat does not define mailbox limits, slow-consumer backpressure, task
  cancellation, graceful shutdown, or atomicity between delivery and history;
- mimichat has a protocol version value but no compatibility negotiation or
  rejection policy.
- mimi-markdown escapes fenced code and document titles but inserts ordinary
  inline text, link destinations, and language-class values without a complete
  escaping or URL-scheme policy;
- mimi-markdown has tests named for images and nested formatting without a
  corresponding complete parser, accepts unterminated fences to end-of-file,
  and its CLI entry point currently prints usage without converting a file;
- mimi-log duplicates every status-filter match and computes `error_rate` with
  an expression whose precedence does not match the stated aggregate;
- mimi-log advertises Nginx latency and JSON decimal parsing while both helpers
  return `0.0`, and format detection never selects Nginx;
- mimi-log hand-builds CSV/JSON without field escaping, silently ignores invalid
  CLI options, and advertises time-range filtering that is not implemented.

Surfacing these questions is a successful MimiSpec outcome: the transcription
made implicit implementation choices reviewable without claiming they were
already human-confirmed requirements.

## Readiness judgment

The result supports a narrower claim than “0.3 is complete”:

- **Yes:** the current tree is useful for real intent capture, structured
  review, exact source preservation, round-trip parsing, and gap discovery.
- **Not yet:** it is not a released 0.3, a production MIMI adapter, or an
  Evidence system. The technical large-review workflow now exists, but still
  needs independent-author validation.

These four AI-assisted reverse transcriptions are technical corpus additions.
They must not be counted as independent authors or documents toward the
external usability release gate.
