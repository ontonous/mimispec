# MimiSpec 0.3 Core 正式设计

> 当前发布版本：`v0.2.1`
>
> 本文状态：MimiSpec `0.3.x` 语言设计基线。当前 main 工作树已经实现本文的
> Core parser/AST/lossless/commitment-slot 主干，但尚未形成正式 0.3 发布；
> crates.io、Cargo 版本和 tag 仍是 `0.2.1`。

## 1. 0.3 的设计中心

MimiSpec 是供人类与 AI 共同阅读和维护的高信息密度意图语言。它保存：

- 最初为什么要做某件事；
- 什么行为必须发生；
- 什么行为不得发生；
- 哪些关系已经确认；
- 哪些内容仍待审或交给 AI 补充；
- 状态、条件、步骤和约束之间的逻辑结构。

0.3 的核心任务不是增加目标语言关键字，也不是建立代码生成、验证或发布
协议，而是修正 0.2.1 中会妨碍意图阅读和保存的语义不一致：

1. `desc` 被错误地隐含解释为“交给 AI 实现”；
2. 同一 scope 中多个 `desc` 会因出现顺序改变含义，甚至被忽略；
3. `requires` / `ensures` 只能保留一条，重复条款会被覆盖；
4. 文档根不能可靠承载匿名上下文中的条件与保证；
5. Flow 缺少事件身份，且没有说明未列出的边究竟是禁止还是尚未表达；
6. 语义 AST、原始文档、段落和后缀槽位尚未形成一致模型。

因此 0.3 的主题是：

> **让一小段独立 MMS 在没有重复外部结构的情况下，仍能精确保存描述、约束、
> 条件、保证、步骤、状态关系和协作状态。**

外部语言或文档容器未来可以把一段文本交给 canonical MimiSpec parser；外部
包裹语法、宿主 AST 和执行语义不属于 MimiSpec Core。

## 2. 语言边界

MimiSpec 0.3 只定义：

- `.mms` 文本的词法和语法；
- 文档根与 Fragment 的意图语义；
- `desc`、`rule`、`requires`、`ensures`、Flow 和 Steps；
- commitment 后缀及人类/AI 修改权限；
- paragraph、attachment、source span 和 lossless document；
- canonical parser、AST、renderer 和诊断的可观察合同。

0.3 Core 不定义：

- MIMI、Rust、TypeScript 或其他目标语言语法；
- `mms {}` 或其他外部包裹的解析实现；
- 从 MMS 自动生成、修改或验证生产代码；
- Target Profile、Materialization、Evidence、Release Scope 或 OSE 工作流；
- 自然语言 `rule` 到形式公式的自动等价关系。

这些能力以后可以消费 0.3 parser 的结果，但不得反向定义 Core 语义。

## 3. 五个语言不变量

### 3.1 自然语言一等

自然语言不是低级逃生口。`desc` 和 `rule` 是 MimiSpec 最稳定、最低门槛的
表达形式。

### 3.2 Fragment-first

不完整不是错误。一行描述、一个约束组、一条条件、一个 Flow 或一个 Steps
都可以独立解析。Fragment-first 不意味着 parser 可以把无法识别的保留结构
静默降级成普通 action。

### 3.3 Context-first root

每份 MMS 文档首先是一个匿名的意图上下文。作者不必为了让一小段 MMS 合法，
虚构或重复 `module`、`func`、`type`、`flow` 名称。

### 3.4 段落与顺序有意义

空行决定 `rule` 前置附着是否继续；多个描述、条款和规则的顺序必须保留。
formatter 不得改变 attachment。

### 3.5 协作状态不改变领域含义

`$`、`?` 等后缀只描述内容当前的人类-AI协作状态。它们不表示代码已经生成、
测试、验证或发布。

## 4. 文档根是匿名 Intent Context

### 4.1 语法

```ebnf
Document        ::= Import* ContextItem*
ContextItem     ::= Desc
                  | Rule
                  | ContextClause
                  | Fragment
ContextClause   ::= RequiresClause | EnsuresClause
```

例如以下内容本身就是完整、合法的 MMS：

```mimispec
desc$ "只有确认外部结果后才提交新状态"
rule$ "结果未知时不得把重试当作一次新的业务操作"

requires: amount > 0
requires: request.idempotency_key != ""
ensures: committed == true or original_state_preserved == true
```

它不需要外层 `func Pay:`。parser 必须把：

- `desc` 解析为当前 Context 的描述；
- `rule` 解析为当前 Context 的环境约束，除非它明确前置附着于后续语义项；
- `requires` / `ensures` 解析为 Context Clause；
- 其他命名结构解析为普通 Fragment。

保留关键字出现在 Context 根时不得通过 fallback 被识别成普通 action。

### 4.2 Context 与外部使用

Core 不知道 Context 最终描述文件、函数、状态、转换、组件还是人工过程。调用方
可以保留匿名 Context，也可以在自己的数据模型中把它与某个外部对象关联。
这种关联不改变 MMS AST 内部含义。

## 5. `desc`：描述意图，不隐含委托

### 5.1 基本语义

```mimispec
desc "支付确认后订单才能进入 Paid"
```

表示作者对当前 scope 或当前步骤的自然语言描述。无后缀 `desc` 不表示：

- 内容是空白；
- AI 自动拥有决定权；
- 必须生成某个代码节点；
- 当前描述尚未确认。

委托和审阅由后缀显式表达：

```mimispec
desc?  "失败后是否进入人工处理"
desc?? "由 AI 提出恢复策略候选"
desc$  "支付确认后订单才能进入 Paid"
```

### 5.2 多个描述

同一 descriptive scope 中可以出现任意多个 `desc`。它们全部保留原始顺序和
各自的 commitment，不允许：

- 只把第一条保存为描述；
- 把后续描述自动转成 Step；
- 在 `type` 中忽略后续描述；
- 合并后丢失单条后缀或 source span。

是否为步骤只由语法上下文决定：

```mimispec
func Pay:
    desc "处理一次支付请求"
    desc "业务提交权威仍是订单系统"
    steps:
        desc "校验请求身份"
        desc "向支付机构发起请求"
```

前两条描述 `Pay`；后两条是自然语言 Step。

## 6. `rule`：约束与段落附着

### 6.1 两种合法归属

`rule` 只有两种结果：

1. 与后续非 `rule` 语义项处于同一段落时，作为前置约束附着该项；
2. 被 ParagraphBreak 阻断、位于 scope 末尾或没有可附着语义项时，成为当前
   scope 的环境约束。

所有可在当前 scope 中独立出现的非 `rule` 语义项都可以接收前置约束，包括
Desc、Clause、Fragment、Field、FlowEntry、FlowArm、Step 和 Placeholder。这样
作者不需要记忆一份“只有某些 AST 节点能接 rule”的例外表。

`ParagraphBreak` 只由至少一个真实空行产生；“真实空行”是除空白字符外没有
其他内容的物理行。独占一行的注释属于文档 trivia，它本身不制造
ParagraphBreak；注释前后若有真实空行，则由该空行终止附着链。

合法 scope 的 lowering 算法固定为：

1. 按唯一 item 顺序扫描；
2. 连续 Rule 进入 pending prelude；
3. pending 后遇到 ParagraphBreak，全部归入当前 scope Environment；
4. 否则遇到下一个非 Rule item，全部 Attached 到该 item；
5. scope 结束仍 pending，全部归入当前 scope Environment。

“只有两种结果”针对语法有效的文档。错误恢复期间可以暂存
`UnresolvedByRecovery`，但必须同时产生诊断并把 Document 标记为 partial；它
不是第三种合法 attachment，也不能被当作 Environment 静默掩盖。

因此单独的规则文件合法：

```mimispec
rule "所有未知结果必须保持可见"
rule "不得用默认值伪造外部资源"
```

它们不是 dangling rule。

### 6.2 附着示例

```mimispec
rule "支付必须幂等"
func Pay:
    ...
```

规则附着 `Pay`。

```mimispec
rule "所有敏感数据必须加密"

func Pay:
    ...
```

规则是当前 scope 的环境约束，不附着 `Pay`。

### 6.3 约束强度

`rule` 表示不可静默违反的意图约束，但其自然语言内容不自动获得形式化语义。
`rule$` 表示这条约束内容已确认，不表示它已经由程序验证。

## 7. `requires` 与 `ensures` 是可重复条款

### 7.1 列表语义

同一 scope 可以拥有零到多条 `requires` 和 `ensures`：

```mimispec
func Pay(order, amount):
    requires: order.status == Pending
    requires$: amount > 0
    ensures: order.status == Paid or order.status == Pending
    ensures$: audit.recorded == true
```

每条 clause 必须分别保存：

- clause kind；
- condition；
- keyword commitment；
- condition/value commitment；
- source span；
- attached rules。

多条 `requires` 默认合取，多条 `ensures` 默认合取。需要选择关系时在单条
condition 中显式使用 `or`。

### 7.2 AST 不变量

`requires` / `ensures` 必须是有序列表，不能用 `Option<Condition>` 或单一
commitment 字段表示。重复条款不得覆盖、合并或静默丢失。

### 7.3 Context Clause

Document 根出现的条款必须作为 Context Clause 保存，而不是 Action Step：

```mimispec
requires: source_is_authenticated == true
ensures: failure_is_observable == true
```

调用方是否把这些条款用于某个外部对象，是调用方自己的关联关系；Core 只保存
其逻辑结构。

## 8. Flow：意图级事件图

### 8.1 定位

MimiSpec Flow 表达作者关心的状态、事件、条件和期望结果。它不是任何目标语言
Flow AST 的缩写，也不描述资源布局、执行后端或运行时调度。

### 8.2 开放世界

Flow 默认是局部、开放的意图图：

- 写出的边是作者明确表达的关系；
- 未写出的边表示“本 Fragment 没有说明”；
- 未写出的边不自动变成允许，也不自动变成禁止；
- `$` / `$$` 表示已写内容的协作状态，不表示图已经穷举完整。

如果作者要表达闭合图，应使用明确规则：

```mimispec
rule$ "未列出的状态事件组合均不允许"
flow$ OrderLifecycle:
    ...
```

### 8.3 可选事件标签

0.3 为 Flow edge 增加可选事件：

```mimispec
flow OrderLifecycle:
    Pending:
        on CaptureConfirmed >>> Paid: desc "支付机构确认扣款"
        on CancelAccepted >>> Cancelled: desc "取消请求已接受"
    Paid:
        on TrackingAssigned >>> Shipped: requires: tracking != ""
```

旧写法继续合法：

```mimispec
flow OrderLifecycle:
    Pending >>> Paid: desc "支付成功"
```

它表示作者只关心状态关系，暂未给事件命名。

目标 grammar：

```ebnf
FlowEntry       ::= State FlowArm
                  | State ":" NEWLINE INDENT FlowArm+ DEDENT
FlowArm         ::= EventLabel? ">>>" Target ":" FlowTail*
EventLabel      ::= "on" (Ident | String)
FlowTail        ::= RequiresClause | Desc
```

Event label、target、guard 和描述分别拥有 commitment slot。

### 8.4 匿名 Context Flow

在匿名 Context 中，作者可以省略 Flow 名称：

```mimispec
flow:
    Pending:
        on CaptureConfirmed >>> Paid: desc "确认后提交"
```

匿名 Flow 只表示当前 Context 的状态意图，不产生虚构名称。命名 Flow 仍用于
独立文档、复用和引用。

## 9. Commitment 仍是组合状态机

0.3 保留九种表层状态以及“先锁、后疑”的组合规则：

```text
无后缀  ?  ??  $  $?  $??  $$  $$?  $$??
```

规范解释以 [`commitment-state-machine.md`](commitment-state-machine.md) 为准。

本轮补充以下边界：

1. `desc` 无后缀不再隐含委托；
2. 每条重复 clause 都有独立状态；
3. Flow event 是独立 value/reference slot；
4. attachment 是独立结构槽位；
5. `$` / `$$` 只表示意图确认和内容保护；
6. commitment 不授予任何目标工具生成、发布或验证权限。

规范术语使用“已确认（confirmed）”。现有 API 名 `is_commit_ready()` 可以在
兼容期保留，但不得被解释为代码、构建或发布 readiness。

## 10. Lossless Document 与 Semantic AST

0.3 必须明确分成两层：

```text
Lossless Document
    原始 token、注释、空行、缩进、后缀位置、完整 span

Semantic AST
    Context、Fragment、Desc、Rule、Clause、Flow、Step 与 attachment
```

### 10.1 scope 内只有一个权威顺序

每个语义 scope 必须拥有一个按源码顺序保存的 item 序列。`descs()`、
`requires()`、`ensures()`、`environment_rules()` 等按类型查询只能是这个序列和
attachment 关系的派生视图，不能成为彼此分离的权威存储。

```text
ContextBody
    items: [Desc, Rule, Clause, Fragment, ...]   // 唯一权威顺序
    attachments: [Rule -> Item | Rule -> Scope] // 显式关系
```

规则被附着后仍保留在原来的有序 item 序列中，不得为了方便查询而移动进目标
节点并丢失它在 Context 中的位置。实现可以提供 `target.attached_rules()` 之类
的反向查询，但它必须从 attachment 关系派生。

这一不变量同时解决：

- `desc`、rule、clause 和 Fragment 交错时的叙述顺序；
- 连续 rule prelude 的原始位置；
- formatter 和 structured move 的稳定 attachment；
- 同一语义事实被多个 Vec 重复保存后发生漂移的问题。

语义 item/slot ID 在一次文档 revision 内稳定即可；0.3 Core 不把它冒充跨修订、
跨文件或跨项目的永久身份。

### 10.2 两层共同不变量

必须满足：

- parser 保留每个非空和空白结构的源范围；
- `ParagraphBreak` 是明确节点或 token，不再通过 newline 数量猜测；
- semantic lowering 产生确定的 rule attachment；
- formatter 不改变 rule target、clause 顺序或 suffix slot；
- structured move 携带 attached rule prelude；
- normalization renderer 不得被描述成原文恢复；
- parse-render-parse 必须保存 semantic AST，但只有 lossless edit 承诺原文保真。

## 11. Canonical Parser 合同

0.3 保留 0.2.1 的三个基础入口，并增加无损入口：

```rust
parse(source)          // Document / anonymous Context
parse_fragment(source) // 单个 Fragment 或 Context Item
tokenize(source)
parse_lossless(source) // 原文 + Semantic AST + span/slot/attachment
```

并满足：

1. `desc` 在根部解析为 Desc Context Item，不再包装为 Steps；
2. 根部 `requires` / `ensures` 解析为 Clause，不再 fallback 为 Action；
3. 重复 clause 全部保存；
4. 匿名和命名 Flow 均可解析；
5. event-labelled Flow edge 可解析；
6. parser recovery 不得用“成功”掩盖保留关键字的错误分类；
7. `ParseResult` 含错误时，partial AST 必须继续明确标记为 partial；
8. semantic parse 与 lossless parse 必须产生相同的 item 分类和 attachment；
9. JSON shape 在 0.3 发布前建立版本字段和迁移说明。

外部调用者只需把 MMS 文本交给 `parse`；它不需要伪造一个命名 Fragment。

`parse_fragment` 是兼容保留的历史名称。0.3 中它解析一个 **Context Unit**：

- 一个非 `rule` Context/body item，以及它同段的前置 rule chain；或
- 一个在 unit 末尾归入 Environment 的 rule chain。

它必须使用与 `parse` 相同的 ContextItem 分类、ParagraphBreak 和 attachment
算法；输入含第二个无关 unit 时返回结构化诊断，不能静默忽略尾部内容。

## 12. 兼容策略

### 12.1 语法兼容

- 所有合法 0.2.1 表层写法继续可解析；
- 旧 Flow 无事件标签写法继续合法；
- 单条 `requires` / `ensures` 含义不变；
- commitment 后缀拼写不变；
- `rule` 的前置附着与 ParagraphBreak 原则保留。

### 12.2 语义修正

以下属于 0.3 有意修正：

- 后续 `desc` 不再自动变成 Step 或被忽略；
- 重复 clauses 不再“最后一条获胜”；
- 根部 clauses 不再被当作 action；
- scope 末尾 rule 明确是环境约束；
- rule prelude 不再受 AST 节点类型白名单限制；
- comment-only line 不再冒充 ParagraphBreak；
- scope 内容不再拆成多个会丢失跨类型顺序的权威 Vec；
- Flow 未列边明确采用开放世界解释。

### 12.3 API 迁移

实现时预计需要：

- `desc: Option<Desc>` 迁移为有序描述集合或等价 first-class item；
- `requires/ensures: Option<Condition>` 迁移为 clause 列表；
- `Fragment::Desc` / `Fragment::Clause` 或等价 Context Item；
- 每个 scope 使用唯一有序 body item 序列，typed collection 仅作查询视图；
- `rule` 保留为 item，并用显式 Attached/Environment 关系记录归属；
- `FlowDef.name` 支持匿名；
- `FlowArm.event` 与对应 commitment/span；
- versioned CLI JSON。

## 13. 0.3 明确不做

1. 不复制 MIMI 的 Fault、Actor、Protocol、Session、Capability 或 FFI 语法。
2. 不把 `rule` 自动翻译为可执行或可证明合约。
3. 不把 commitment 与实现、测试、验证或发布状态合并。
4. 不要求作者为了合法片段补写虚构名称和容器。
5. 不增加 Profile、Materialization、Evidence 或 Release Scope 到 Core AST。
6. 不让 parser 根据外部宿主类型改变同一段 MMS 的内部语义。
7. 不用教程、CLI 或 IDE 行为替代语言规范。

## 14. 冻结门禁

0.3 Core 只有满足以下条件才可冻结：

- `desc`、`rule`、Context Clause 和 Flow 的规范无互相矛盾表述；
- 多描述、多 requires、多 ensures 的语料没有信息丢失；
- 根 Context 与命名 Fragment 使用同一 parser；
- event-labelled 与旧式 Flow 均能 round-trip；
- Flow 开放性不从 commitment 或缺少 `...` 推断；
- paragraph 和 attachment 经格式化保持一致；
- scope 的跨类型 item 顺序保持一致；
- comment-only line 保持 attachment，真实空行精确切换 attachment；
- keyword/name/value/event/attachment 后缀槽位均有测试；
- 0.2.1 兼容语料通过；
- 至少使用状态、失败恢复、资源权限、通信和外部边界五类现实意图进行冷读；
- 不熟悉目标语言的读者只看 MMS 仍能说清目标、禁区和未决问题。

## 15. 总结

MimiSpec 0.3 的核心循环是：

```text
描述当前意图
-> 声明不可违反的规则
-> 用条件、保证、步骤或 Flow 增加必要逻辑
-> 用 ? / ?? 标出审阅与委托
-> 由人类确认并使用 $ / $$ 保护
-> 在原始文档结构不被破坏的前提下继续演进
```

0.3 的成功不取决于生成了哪种代码，而取决于同一段 MMS 是否能被人类和 AI
一致理解、完整解析、可靠编辑，并长期保留最初意图。
