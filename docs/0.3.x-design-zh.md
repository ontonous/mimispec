# MimiSpec 0.3.x 关键设计总纲

> 当前已发布版本：`v0.2.1`
>
> 本文状态：`0.3.x` 系列设计基线

## 1. 0.3.x 的核心任务

0.2.1 已经证明 MimiSpec 可以稳定解析 Fragment、后缀、约束链、Flow、
Steps、UI 和跨文件引用。0.3.x 不应优先继续增加表层关键字，而应回答一个
更根本的问题：

> 人类和 AI 如何通过同一份 `.mms`，持续推动每个意图槽位从未知、委托、
> 审阅，走向锁定、物化和基于新证据的再次审视？

因此 0.3.x 的主题是：

> **把已经存在的 `$`/`?` 后缀，从可解析元数据升级为可执行的人类-AI
> 协作协议。**

这不是把 MimiSpec 变成编程语言。五分钟入门承诺必须保持：没有编程经验的
人只学会 `desc`、`rule` 和 `??`，就可以开始真正工作。

## 2. 语言定位

MimiSpec 是接近 Markdown 的通用意图语言：

- 自然语言是一等内容；
- Fragment 可以独立存在；
- 不完整不是错误；
- 空行具有文档段落意义；
- 人类不需要先学会类型、状态机或形式化验证；
- AI 和工具承担结构化、检查和目标语言映射的复杂度。

MimiSpec 与 Mimi 是两门相对独立的语言。两者拥有独立语法、AST、工具链、
版本和发布周期，也都可以在不安装对方的情况下独立使用。

在这个独立性前提下，Mimi 是 MimiSpec 的原生和第一方生产目标：

- 两者共同演进；
- Mimi Profile 应拥有最深的语义覆盖和验证能力；
- MimiSpec Core 不绑定 Mimi AST，因此同一份 MMS 仍可约束 Rust、
  TypeScript、Python 等语言。

“原生目标”不表示 MimiSpec 是 MIMI 前端，也不表示所有 MMS 必须经过 MIMI。
它只表示官方会优先维护一个外部 Mimi Profile，使已锁定意图能够更深入地映射
到 MIMI，并利用 MIMI 工具链返回更强 Evidence。

可以概括为：

```text
MimiSpec Core：通用人类意图
Mimi Profile：第一方、默认、最深的物化解释
其他 Profile：诚实声明能力边界的通用目标适配器
```

MIMI 中现存的 `mms {}` 不是这套生产桥接：它只是历史性文本嵌入/超级注释，
不调用 canonical MimiSpec parser，不执行 commitment，不参与类型检查、解释、
codegen 或验证。真正的生产映射必须由独立、版本化的 Mimi Profile/adapter
完成。

## 3. 后缀状态机

后缀并不是九个互不相关的标签，而是组合运算。

当没有锁时，`?`/`??` 描述内容：

```text
A?  = 对 A 的内容不确定，请 AI 提供选项或审视
A?? = A 的内容交给 AI 决定
```

当已有 `$`/`$$` 时，`?`/`??` 描述锁定判断：

```text
A$?   = 人类认为 A 可能可以锁定，请 AI 判断锁是否成熟
A$??  = 普通锁成熟度委托 AI 评估；内容保持受保护，结果进入人类审阅
A$$?  = 人类认为 A 可能可以强锁，请 AI 判断是否真的达到强锁条件
A$$?? = 强锁成熟度委托 AI 评估；内容保持受保护，结果进入人类确认
```

所以：

```text
A$? = Question(Lock(A))
```

而不是：

```text
Lock(Question(A))
```

这也是后缀必须“先锁、后疑”的原因。

## 4. 九种协作状态

| 后缀 | 状态 | 核心含义 |
|------|------|----------|
| 无 | 开放草案 | 当前内容可继续协作优化 |
| `?` | 内容待审 | 人类有方向，需要 AI 给出候选或判断 |
| `??` | 内容委托 | 内容决策交给 AI |
| `$` | 普通锁定 | 内容和普通锁判断均已确认 |
| `$?` | 普通锁待审 | 内容受保护，锁定成熟度待审 |
| `$??` | 普通锁委托评估 | 内容受保护，AI 评估后转入普通锁人类审阅 |
| `$$` | 强锁定 | 内容和强锁判断均由人类确认 |
| `$$?` | 强锁待审 | 内容受保护，强锁成熟度待审 |
| `$$??` | 强锁委托评估 | 内容受保护，AI 评估后请求人类确认强锁 |

所有包含 `$` 的状态都保护内容。`?`/`??` 打开的是关于锁的讨论或评估，不是
内容编辑权。Tooling 只能代理 Human 或 AI，不拥有独立的状态转移权限。

普通锁保护节点身份、名称、种类、位置、顺序、容器结构和 attached rule
prelude；显式 `?`/`??` 子槽位仍可按自身状态演进。强锁默认保护整个现有结构
子树，包括无后缀子槽位；只有显式 `?`/`??` 子槽位保持开放。

## 5. AI 对锁定的挑战

AI 可以执行：

```text
$  -> $?
$$ -> $$?
```

其含义是：

> AI 不修改内容，只认为现有锁定判断可能需要重新审视。

这个操作必须满足：

1. 内容不变；
2. 名称不变；
3. 节点位置和顺序不变；
4. `rule` 附着对象不变；
5. `$` 数量不减少；
6. 必须附带理由和证据；
7. 被人类拒绝的同一挑战，除非出现新证据，否则不能反复提出。

AI 不能执行：

```text
$  -> 无后缀
$$ -> $
$$ -> 无后缀
任意状态 -> $$
```

强锁的最终确认和解除必须由人类完成。

## 6. 后缀作用于语义槽位

```mimispec
func?? Pay$:
    desc "处理支付"?
```

这里不是一个状态，而是三个状态：

- `func??`：实体形态交给 AI 判断；
- `Pay$`：名字已经锁定；
- `"处理支付"?`：描述内容需要 AI 提供更好的候选。

因此 0.3.x 的 Parser 和无损 AST 必须分别保存：

- 关键字后缀；
- 标识符后缀；
- 字符串和值后缀；
- 每个后缀的 Span；
- 后缀实际控制的语义槽位。

Formatter 不得把后缀从一个槽位移动到另一个槽位。

## 7. `rule` 与段落

空行附着语义是 MimiSpec 的核心自然文档能力，不应移除。

```mimispec
rule "支付必须幂等"
rule "所有扣款必须有审计记录"
func Pay:
```

连续规则构成约束前奏，并附着给紧随其后的 `Pay`。

```mimispec
rule "所有敏感数据必须加密"

module Payment:
```

空行终止附着链，使规则成为当前层级的环境约束。

0.3.1 必须将空行提升为明确的 `BlankLine` 或 `ParagraphBreak`，不能继续只靠
若干 `Newline` 的数量猜测，因为：

```text
空行 = 语义
```

Parser、Formatter、structured edit 和 IDE 都必须保证规则附着稳定。

## 8. 文档是状态机森林

整个文件没有唯一的“草案”或“锁定”状态。真实文件可能同时包含：

```text
项目目标          $$
支付规则          $?
退款规则          ?
数据库选择        ??
Pay 名称           $
Pay 步骤           无后缀
```

所以文档状态是所有语义槽位状态的组合，而不是一个线性枚举。

OSE 和 IDE 应从这些局部状态生成任务队列：

- 决策队列：`?`、`$?`、`$$?`；
- 委托队列：`??`、`$??`、`$$??`；
- 锁定挑战队列：AI 发起的 `$ -> $?`、`$$ -> $$?`；
- 物化队列：当前 Release Scope 内选中的 `$`、`$$`。

传统的 Draft、Review、Lock、Commit 可以保留为产品阶段摘要，但不能再作为
语言的真实状态模型。

## 9. Commit 与物化

只有 `$` 和 `$$` 可以作为已确认意图物化：

```text
CommitReady = {$, $$}
```

以下状态不能 Commit：

```text
?
??
$?
$??
$$?
$$??
```

但 Commit 不等于“整份文件已经结束”，也不等于“实现已经被证明”。它只是：

> 把选定范围内已经确认的意图投影到某个目标世界。

0.3.5 必须支持局部物化和 Release Scope。正式发布只要求本次范围内的必要
槽位 100% 锁定，而不是强迫未来规划和扩展槽位全部锁定。

## 10. 锁定与证据分离

`$` 回答：

> 人类是否确认了这个意图？

Evidence 回答：

> 某个目标产物是否满足这个意图？

二者必须分开。一个 `$` Fragment 可以：

- 尚未实现；
- 已实现但未测试；
- 测试失败；
- 在 MIMI 中完成类型检查和验证；
- 与生产代码发生漂移。

证据通过侧车数据或 OSE 项目索引保存，不继续增加源码后缀。

## 11. MIMI 原生 Profile

MIMI Profile 应当利用 MIMI 的 Flow、Fault、Recover、Capability、Protocol、
Session 和验证工具提升意图理解，但不能要求普通用户学习这些机制。

用户写：

```mimispec
rule$ "扣款成功但订单更新失败时，不能重复扣款"

func$ Pay:
    steps:
        check balance
        charge payment
        update order status
```

MIMI Profile 可以发现：

- 需要中间状态；
- 需要幂等标识；
- 需要失败恢复选择；
- 可能需要 persistent 数据和 Fault/recover；
- 需要重复事件测试。

但它应当用自然语言向人类提问，而不是要求人类写 MIMI 级状态机。MIMI 的
技术结构属于 Profile 的物化结果，并标记为目标派生内容。

## 13. 0.3.x 明确不做的事情

1. 不把 MIMI 的全部语法复制进 MimiSpec。
2. 不让自然语言规则自动冒充已经证明的形式公式。
3. 不允许 AI 自动锁定或强锁内容。
4. 不要求整份文档锁定后才能进行局部物化。
5. 不让生产代码成为与 MMS 竞争的事实源。
6. 不在真实 OSE 协作验证前宣布 1.0 语法冻结。

## 14. 总结

0.3.x 完成后，MimiSpec 的核心循环应成为：

```text
表达
-> 委托或提问
-> AI 提供候选
-> 人类确认
-> 锁定候选
-> 锁定
-> 物化
-> 获得实现证据
-> 保持确认或重新审视锁定
-> 继续演进
```

MimiSpec 不只是保存最终规格，而是保存整个软件意图当前所处的协作状态。
`desc` 和 `rule` 负责表达意义，`$` 和 `?` 负责驱动意义在人类与 AI 之间
持续收敛。
