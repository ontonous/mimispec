# MimiSpec 0.3.0-dev Usability Readiness Report

> Date: 2026-07-20
>
> Scope: the unreleased `main` working tree, which is technically ready to be
> cut as a `0.3.0-dev` development snapshot. The published crate, Cargo
> version, and release tag remain `0.2.1`; no dev artifact is claimed here.

## Conclusion

The `0.3.0-dev` technical snapshot is complete enough for explicit
source/binary evaluation. It is **not an RC or stable 0.3 release**.

- Core milestones M0-M3 are implemented against the current tracked
  design and pass the parser, lossless-document, collaboration, diagnostic,
  corpus, and JSON gates.
- M4 now provides a long-running stdio LSP, advisory/strict authoritative
  sessions, UTF-16 edits, semantic tokens, effective-protection hover,
  rule/Flow navigation, and actor-declared edit transactions under the frozen
  `mimispec.ls/0.3` wire schema.
- Technical M5 stabilization includes corpus, property, performance,
  conformance, migration, API/wire freeze, and package gates. The independent
  5-author/25-document trial remains incomplete, so the project has not cut a
  0.3 release candidate or changed the published version/tag.

This means the current tree may be described as a `0.3.0-dev` snapshot
candidate for continued Core-language testing and integration work. It must
not be described as an already published artifact, an RC, or a stable 0.3
toolchain.

The follow-up [real-project transcription trial](0.3-real-project-transcription-report.md)
translated four non-trivial MIMI projects into four reviewable MMS documents.
All pass lossless/round-trip gates. Their Action
recovery, review-queue scaling, and nested-diagnostic findings now have
technical implementations; the Mimi Profile/provenance boundary remains
experimental and independent-author validation is still outstanding.

## Real-world fixture

[`corpora/real-world-family-ledger.mms`](corpora/real-world-family-ledger.mms)
is a cohesive local-first family-ledger specification rather than an isolated
syntax sample. It covers:

- plain-language product intent and AI delegation;
- locked privacy, offline-first, idempotency, and audit rules;
- record and enum types;
- an event-labelled synchronization Flow with failure and retry paths;
- three contracted functions with ordered and conditional steps;
- two UI views for quick entry and monthly review;
- the distinction between local-save success and synchronization success.

The fixture deliberately uses names such as `Decimal`, `Timestamp`, and
`HouseholdMember` without pretending that the current flat symbol index is a
semantic type checker. MimiSpec accepts progressively incomplete intent; the
absence of name/type errors is not proof that an implementation exists.

## Automated usability contract

`corpus_real_world_family_ledger_is_usable_end_to_end` asserts all of the
following:

1. semantic parsing completes with no error;
2. lossless parsing preserves the exact source bytes;
3. normalization render and reparse preserve the semantic AST;
4. AST JSON carries `mimispec.ast/0.3`;
5. the document retains 3 types, 1 Flow, 3 functions, and 2 UI views;
6. diagnostics expose exactly the two intentional AI delegations;
7. no syntax, rule-attachment, or false missing-failure-path diagnostic is
   emitted.

The fixture is embedded with `include_str!`, so removal, renaming, or a syntax
regression fails `cargo test --lib` in CI.

## Bug found by the fixture

The initial diagnostic run reported that `ExpenseSync` had no obvious
failure/cancel path even though it contains the event `UploadFailed` and a
retry state. The heuristic inspected only state and target names, not Flow
event labels.

The diagnostic now checks source state, event label, and target state. It
recognizes common English and Chinese failure/cancel terms, with a regression
test for an event-labelled failure path. This is a heuristic intent hint, not
formal Flow completeness proof.
