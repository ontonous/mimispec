# MimiSpec 现实工程意图对齐深度评估计划

## 目标

通过亲自编写、解析和审查一组来自现实工程问题的 MimiSpec，重新理解 MimiSpec 0.2.1 已具备的规格表达能力，并评估 MIMI Pre-1.0 的大量稳定裁决是否要求 MimiSpec 0.3.x 修正。

本评估只讨论**意图能否被忠实传递和承接**，不讨论 MimiSpec 是否覆盖 MIMI 的语法、关键字或 AST。MMS 必须先于任何目标语言解释独立成立。

## 核心假设

待检验的主假设：

> MimiSpec 0.2.1 已经能够把现实工程转写成具有目标、约束、结构、行为、风险和未决决策的有效规格；0.3.x 的主要职责应是加强这些意图在人-AI协作中的状态、保护、来源、演进和证据，而不是增加目标语言语法投影。

竞争假设：

> 现实工程中的关键意图无法在 0.2.1 中自然保存，必须修改 MimiSpec Core 的表达模型，而不只是增强 0.3.x 的协作与工具层。

## 方法红线

1. 不在 MMS 中使用 `generation`、`PeerFault`、`Component IR`、`Verified Core` 等 MIMI 专属机制，除非现实工程作者本身明确把它们作为需求。
2. 不建立 MimiSpec Fragment、关键字或 AST 与 MIMI 构造的一一对应表。
3. 不用“可以生成某段 `.mimi`”证明意图表达成功。
4. 不因 MIMI 无法承接某项意图就判定 MMS 无效；这首先是目标能力缺口。
5. 不因 MIMI 有某种机制就把该机制反向写成用户意图。
6. 不把 `$`/`$$` 当作实现完成、目标稳定或验证通过。
7. 0.2.1 与未发布 0.3.x 必须分栏评估，不能用开发线能力替 0.2.1 作证。

## 评估样本

选择一个现实但目标语言无关的“订单支付可靠性”工程问题。它不是 MIMI 示例，而是常见业务系统问题，至少包含：

- 用户和业务希望达成的结果；
- 不可违反的业务约束；
- 外部支付机构、订单存储等现实边界；
- 正常路径与部分失败后的可观察结果；
- 重试、重复请求、并发、超时和人工介入风险；
- 隐私、审计和可运营性要求；
- 尚未由需求方决定的问题；
- 已确认内容与可委托内容。

样本输入必须以自然语言工程叙述开始，不从 MIMI 规范或语法反推。

## 产物

在一个隔离的评估目录中生成以下文件，避免混入正式规范：

1. `problem.md`：现实工程原始叙述和事实来源。
2. `intent-baseline-0.2.1.mms`：只使用 MimiSpec 0.2.1 已发布语法的目标无关规格。
3. `intent-collaboration-0.3.mms`：保持相同领域含义，仅使用 0.3.x 协作能力表达审阅、委托、锁定保护和局部物化状态。
4. `intent-ledger.tsv`：逐项记录原始意图、MMS 位置、表达方式、保真结论、歧义和目标无关性。
5. `mimi-alignment.tsv`：后验记录 MIMI Pre-1.0 对每项意图的承接方式、需澄清项、能力缺口和目标派生义务。
6. `attempts.tsv`：记录每次编写、解析、重写和审查，包括失败尝试。
7. `assessment.md`：最终评估报告和对 MimiSpec 0.3.x 的修正建议。

TSV 日志统一使用制表符，失败和被否决的写法也必须保留，不得只展示最终成功版本。

## 执行步骤

### 1. 建立事实基线

- 从 MimiSpec 0.2.1 发布基线的 README、规范、AST、parser 和测试确认实际可用语法。
- 使用 Git tag `v0.2.1` 或等价发布源码核对，而不是仅依赖当前 main 上的 0.3.x 文档。
- 记录 0.2.1 能表达什么、解析器实际接受什么，以及 renderer/partial AST 的边界。
- 把当前 main 的 0.3.x 能力另列为后续对照，不混入基线。

完成条件：得到一份明确的 0.2.1 可用构造清单，但该清单只用于保证语料可解析，不作为与 MIMI 对齐矩阵。

### 2. 编写现实工程原始叙述

- 以产品负责人、业务人员、运维人员和终端用户能理解的语言描述工程问题。
- 先不出现 MimiSpec 或 MIMI 术语。
- 将内容分为事实、目标、硬约束、期望行为、风险、未决决策和验收观察。
- 明确哪些内容来自需求方，哪些只是评估者假设。

完成条件：不了解 MIMI 的工程团队也能审阅并指出叙述是否符合现实问题。

### 3. 第一次亲写 0.2.1 MMS

- 从最小 `desc`/`rule` 入口开始，不预先追求完整模块树。
- 逐步增加有助于消除歧义的结构，如 module、type、func、flow、steps、requires、ensures、math 或 UI；只有现实意图需要时才增加。
- 每次结构化都在 `attempts.tsv` 记录它解决了什么歧义，若只是为了显得“像代码”则撤销。
- 使用 commitment 后缀表达作者已经给出的不确定、委托和确认状态，但不虚构人类确认。

完成条件：形成 `intent-baseline-0.2.1.mms`，且每个结构都能追溯到原始工程意图。

### 4. 实际解析与往返验证

- 使用 v0.2.1 CLI/library 解析语料，保存真实命令和结果。
- 检查 diagnostics，任何 lexer/parser 错误必须修正或登记为表达缺口。
- 执行 parse-render-reparse，区分语义 AST 往返与原文无损保存。
- 检查 AST 中 `desc`、`rule` 附着、Flow、步骤和 commitment 是否落在预期槽位。
- 不把“解析成功”当作“意图表达成功”，它只是一项最低门禁。

完成条件：0.2.1 语料解析通过；若无法通过，日志必须说明是语法误用、parser gap 还是语言表达缺口。

### 5. 建立逐意图保真账本

`intent-ledger.tsv` 至少包含：

```text
id	source_intent	kind	mms_location	representation	preserved	ambiguity	target_neutral	notes
```

对每项原始意图判断：

- 是否完整保留；
- 是否被结构化时意外收窄；
- 是否被自然语言保留但暂时不能静态检查；
- 是否在 MMS 中失去来源、优先级、责任人或未决状态；
- 是否偷偷引入了 MIMI 专属假设。

完成条件：所有原始意图都有去向，不能遗漏不方便表达的反例。

### 6. 独立领域审查

- 委托一个不了解 MIMI 映射目标的审查者，只阅读 `problem.md` 和 0.2.1 MMS。
- 要求其列出丢失、篡改、过度形式化、仍含歧义和不可验收的意图。
- 审查者不得建议 MIMI 关键字或实现机制。
- 根据审查结果修改 MMS，并在 `attempts.tsv` 保留前后差异和理由。

完成条件：领域审查不再发现高严重度意图丢失；残余歧义被明确标成未决，而不是由作者擅自补全。

### 7. MIMI Pre-1.0 后验承接审查

- 在领域 MMS 冻结后，才让 MIMI 侧审查者读取它。
- 按每个意图判断 MIMI Pre-1.0 是否能完整承接、部分承接、需要澄清或不适合由 MIMI 承担。
- 记录 MIMI 可能采用的机制，但全部标为“目标派生解释”，不得反写成原始人类意图。
- 区分：领域意图缺口、Mimi Profile 理解缺口、MIMI 目标能力缺口、MIMI 当前实现成熟度缺口。
- 检查 MIMI 的 stable 裁决是否使旧物化解释失效，而不是是否迫使 Core 增加同名语法。

`mimi-alignment.tsv` 至少包含：

```text
intent_id	mimi_fit	clarification_needed	derived_obligation	target_requirement	implementation_maturity	profile_gap	must_change_mms
```

完成条件：每项意图都有后验承接结论，并明确 `must_change_mms` 的理由；只有原始意图本身含糊或自相矛盾时才允许为真。

### 8. 用 0.3.x 协作模型重写同一规格

- 不增加新的领域含义，只增强槽位级审阅、委托、锁定挑战、规则附着保护、来源和 Evidence 关系。
- 对比 0.2.1 版本，标出哪些变化只是工具/协议增强，哪些确实改变了语言可表达性。
- 模拟至少三种现实演进：目标语言规范变化、实现证据失效、需求方改变已确认决定。
- 验证这三种变化是否可分别处理，而不混淆 commitment、target status 和 evidence。

完成条件：同一领域意图在两个版本间可逐项对应；不得用 0.3.x 偷偷修补原始需求内容。

### 9. 归因 MimiSpec 0.3.x 是否需要修正

按以下分类输出发现：

1. **0.2.1 已充分表达**：无需修改 Core。
2. **0.2.1 能表达但工具未利用**：改 parser/query/diagnostics/IDE/Profile，不扩语法。
3. **0.3.x 协作模型应增强**：如来源、actor、challenge、drift、Evidence 或段落保护。
4. **Target Profile 应增强**：MIMI 对意图的解释、澄清、能力报告和派生义务。
5. **真正的 Core 表达缺口**：只有目标无关、跨 Profile、无法用自然语言一等与现有结构诚实保存的现实意图才能进入此类。
6. **非 MimiSpec 职责**：目标实现细节、编译器成熟度或 MIMI 专属机制。

每一项修正建议都必须引用 `intent-ledger.tsv` 或 `mimi-alignment.tsv` 的具体 ID，禁止从 MIMI 新增特性直接推导 MimiSpec 新语法。

### 10. 对抗性复核

- 让独立审查者专门寻找“把目标实现机制冒充用户意图”的位置。
- 检查是否把自然语言 `rule` 错误宣称为形式证明。
- 检查是否把 MIMI stable、MMS `$` 和当前 implementation complete 混成一个状态。
- 检查是否只选择了 MIMI 容易承接的意图而遗漏 UI、运营、合规、人工流程或外部事实。
- 检查结论在目标换成 Rust/TypeScript/人工流程时是否仍成立；若不成立，说明 MMS 已被 MIMI 污染。

完成条件：所有高严重度方法论问题关闭，或作为明确反例保留在报告中。

## 评价标准

### 意图保真度

- 原始意图覆盖率：目标 100%，包括无法结构化的内容。
- 高严重度语义篡改：0。
- 未注明的评估者假设：0。
- 每个锁定状态均有真实作者依据，不能由评估者擅自添加。

### 目标无关性

- MMS 中为了 MIMI 而添加的专属机制：0。
- 可更换目标 Profile 而无需改写领域意图。
- MIMI 派生义务全部有独立 provenance。

### 可执行审计性

- 0.2.1 语料实际解析通过。
- 所有尝试和失败均写入 `attempts.tsv`。
- 所有结论可追溯到原始意图 ID 和具体 MMS 位置。
- 版本状态严格区分发布 0.2.1、main 0.3.x、MIMI Pre-1.0 目标与当前 MIMI 实现。

## 最终报告必须回答的问题

1. MimiSpec 0.2.1 是否已经能把现实工程忠实转写为 SPEC？“能”的边界是什么？
2. 自然语言一等和结构化 Fragment 如何共同工作，而不是谁替代谁？
3. 哪些现实信息能表达但目前不能静态检查？这是否构成语言缺口？
4. MIMI Pre-1.0 锁定的大量内容改变的是原始意图，还是对这些意图的目标解释？
5. MimiSpec 0.3.x 真正需要修正的是 Core、协作协议、Evidence、Profile，还是仅文档表述？
6. 哪些先前建议源自错误的语法映射思维，应该撤回？
7. 如果改用 Rust、TypeScript 或人工流程承接同一 MMS，规格是否仍然成立？

## 停止条件

只有同时满足以下条件才结束评估：

- 完成至少一份非玩具现实工程叙述；
- 完成并实际解析一份 0.2.1 MMS；
- 每项原始意图进入保真账本；
- 完成独立领域审查和 MIMI 后验承接审查；
- 完成同意图的 0.3.x 协作版本；
- 完成对抗性复核；
- 对所有 0.3.x 修正建议给出意图级证据；
- 明确撤回或保留此前“因 MIMI Pre-1.0 而修正 MimiSpec”的每一类判断。

## 预期结论形式

不预设 MimiSpec 0.3.x 必须修改或无需修改。最终只接受如下证据链：

```text
现实工程叙述
-> 目标无关的原始意图
-> MimiSpec 0.2.1 表达与解析证据
-> 意图保真审查
-> MIMI Pre-1.0 后验承接/缺口
-> 0.3.x 协作模型对照
-> 分层修正结论
```

任何从“MIMI 新增或锁定了 X”直接跳到“MimiSpec 应增加 X 语法”的结论，一律判为无效。

## 0.3 Core 正式语义收敛（2026-07-18）

用户校准后，本计划不再以 Profile、Materialization、Evidence 或外部 `mms {}`
为 0.3 主线。正式语言设计已经冻结以下新增内部不变量：

1. Document 是匿名 Intent Context；
2. `desc` 不隐含委托，多条描述全部保留；
3. clauses 是有序多值，禁止覆盖；
4. Rule 只有 Attached / Environment；
5. 每个 scope 只有一个权威有序 item 序列，attachment 单独建边；
6. 所有同 scope 非 rule item 都可接收 prelude，不保留 AST 类型白名单；
7. 只有真实空行产生 ParagraphBreak，comment-only line 只是 trivia；
8. Flow 支持匿名和可选 Event，缺省采用开放世界；
9. commitment 只表示意图确认和保护；
10. 外部调用只把原始 MMS 文本交给 canonical parser，不改变 Core 语义。

正式来源为 `docs/0.3.x-design-zh.md`、`docs/specification.md`、
`docs/commitment-state-machine.md` 和 `docs/roadmap-0.3.x.md`。已发布 0.2.1 parser
不具备全部目标语义；当前 main 工作树已于 2026-07-19 进入实现阶段并让目标语料
转绿，但尚未发布、未改 Cargo 版本。

## 当前执行审计（2026-07-18）

现有 `devdocs/mimispec-intent-evaluation/` 已完成 MIMI pre-1.0 权威文档的
40 项意图抽取、0.2.1 语料、发布 tag parser 验证、往返、两轮冷审、0.3
协作摘要、逐项追踪和初版评估。

但当前执行与原计划存在一个重要偏差：实际样本是“从 MIMI 权威文档抽取后
去目标语法化的语言工程意图”，不是第 2 节计划的独立订单支付现实叙述。
因此它可以反驳“目标机制必须反向进入 Core 语法”，但不能独立证明跨产品、
UI、运营、合规、科学计算和人工流程的表达充分性。

继续设计已确认以下未闭合项：

1. `mimi-pre1-intent-v03.mms` 只状态化主题摘要，缺少导入意图的稳定 Slot 身份；
2. revision-local `SourceNodeId` 不能作为跨修订 provenance/Evidence 身份；
3. Explore 候选与 ConfirmedProjection 必须成为不同派生模式；
4. Evidence 必须从节点级工具结果升级为 Claim 级、多来源、可失效记录；
5. Profile 必须支持逐意图组合路由和 residual intent，Mimi 不作为默认单后端；
6. `ConfirmationReady`、`ProjectionReady`、`ReleaseReady` 必须分开。

所以本计划的原始停止条件仍未全部满足。下一轮实证应补做一个不从任何目标
语言规范反推的现实工程样本，并分别用 Rust、TypeScript 与人工流程 Profile
进行后验承接审查；在此之前，当前结论应标为有力反例和设计校准证据，而非
Core 表达完备性证明。

本轮协议收敛已把上述 1-6 项从原则推进为可测试的对象与事务合同，正式草案见
`docs/derivation-evidence-profile-protocol.md`。新增的关键闭包包括：后缀锚点的
semantic footprint 与继承边界、按产物计算影响槽位集合、候选禁止原地晋升、
copy/split/merge 的身份规则、Evidence 的
outcome/validity/policy judgement 分离、组合路由固定点，以及 source/import
修订在长运行 Profile 前后的双重校验。

这关闭的是“设计是否足够指导实现”的一轮缺口，不关闭原计划的跨领域实证
停止条件。支付现实样本、Rust/TypeScript/人工流程后验审查、独立领域冷审和
对抗性复核仍是下一轮证据工作；在它们完成前，不把 sidecar identity、标准
Claim predicate 或默认 Evidence policy 冻结为 1.0 表层契约。

## 正式设计方向修正（2026-07-18）

用户进一步明确：本轮只设计 MimiSpec 0.3 语言与 canonical parser，不设计
任何外部包裹实现，也不以机械约束、代码生成、Profile、Evidence 或 OSE
工作流作为 0.3 Core 的中心。

因此此前关于 Materialization/Profile/Evidence 的工作保留为历史审计材料，
但不再属于本计划的 0.3 Core 停止条件。新的正式证据链是：

```text
现代工程意图压力
-> 0.2.1 短片段与重复结构 parser 实测
-> silent loss / misclassification 证据
-> Context/desc/rule/clause/Flow/commitment 语义裁决
-> lossless document 与 canonical parser 合同
-> 0.2.1 兼容和 0.3 target corpus
```

新增 P0 发现：

1. 普通 `desc` 不得隐含 AI 委托；
2. descriptive scope 中所有 `desc` 必须保留，不能按出现顺序转 Step 或忽略；
3. Document 根必须是匿名 Intent Context；
4. 根 `requires`/`ensures` 不得 fallback 为 Action；
5. 重复 clauses 必须使用有序列表，禁止最后一条覆盖前文；
6. scope-terminal `rule` 是 Environment rule；
7. Flow 默认开放，commitment 不表示图完整；
8. Flow edge 增加通用可选事件 `on Event`；
9. Lossless Document 与 Semantic AST 分层；
10. commitment 只表示意图确认，不定义任何下游生产权限。

正式设计产物改为：

- `docs/0.3.x-design-zh.md`：Core 正式设计；
- `docs/specification.md`：目标表层语法与 AST；
- `docs/commitment-state-machine.md`：后缀与 Actor 权限；
- `docs/roadmap-0.3.x.md`：仅语言/parser/document/language service 路线；
- `docs/migration-0.2-to-0.3.md`：语义与 API 迁移；
- `devdocs/mimispec-intent-evaluation/mimi-pre1-intent-v03.mms`：0.3 验收语料；
- `devdocs/mimispec-intent-evaluation/assessment.md`：正式评估结论。

新的停止条件：

- 正式文档对 Context、desc、rule、重复 clauses、Flow openness/event 和
  commitment 没有矛盾；
- target corpus 覆盖上述新增语义；
- 覆盖矩阵和实验日志记录 current-red 与 target behavior 的区别；
- 后续实现前可以从文档直接写出 parser/AST 失败测试；
- 不再从某个目标语言的新机制直接推导同名 MimiSpec 语法。

## 0.3 Core 实现收口（2026-07-19）

上述停止条件已经从“可写失败测试”推进为当前 main 的实际实现：

- 0.3.0 Context/desc/rule/clause/Flow/commitment 语义已经接入 canonical parser；
- 0.3.1 lossless Document、ParagraphBreak、精确 slot/node/rule span 与 attachment
  已接入；
- 0.3.2 Human/AI transition、protected footprint/hash、lock challenge 与
  semantic-slot patch validator 已接入；
- 0.3.3 complete/partial 状态、`mimispec.ast/0.3`、`mimispec.parse/0.3`、
  Context query 与 recovery contract 已接入；
- 0.3.4 仍是 library-level language-service prototype；
- 0.3.5 已开始兼容和性能稳定化，尚未完成 fuzz/property、跨领域冷读与 API freeze。

本轮新增的性能门禁使用 1000 函数文档同时执行 `parse_lossless()` 和 semantic-slot
快照。旧的逐 token/逐槽位全量扫描超过 60 秒未完成；revision-local token、owner、
parent、position 与 attachment index 使同一 debug 测试在本机约 0.14 秒完成，并由
穷举等价性测试保护结果不漂移。

当前复验基线：171 个 library tests、1 个 CLI test、6 个 doc tests；两份 MMS
语料均返回 complete/零错误的 0.3 版本化 JSON。Cargo、tag 和 crates.io 仍保持
`0.2.1`，这里记录的是未发布 main 能力，不是发布声明。
